Design Rule

CAMEL scouts collect data | v structured findings with access levels | v genome predictors interpret differently | v debate + consensus + bets

The harness should not turn every colony member into a costly free-form LLM worker. CAMEL is best used for fewer, tool-using scouts. Genomes are best for many reproducible opinions.

Genome Predictor

Many cheap simulated decision-makers. Each one interprets the same data through different numeric preferences.

  • Identitypersona, model label, genome hash
  • Inputsstats, odds, news, debate finding
  • Parameterssource weights, risk appetite, edge threshold, herd bias
  • Outputprivate forecast, public claim if selected, final bet or pass
  • Best forscale, reproducibility, analysis, evolutionary loops

CAMEL Scout

Fewer tool-using LLM workers. Each one can search, browse, call APIs, and return structured evidence.

  • Identitysystem message, role, model backend
  • Inputstask prompt, tools, memory, external APIs
  • Parametersmodel config, tools, response schema, memory policy
  • Outputstructured datasource finding with citations and confidence
  • Best fordata collection, extraction, summarization, tool orchestration

Recommended split: CAMEL answers "what evidence exists?" and genomes answer "how should a population of predictors weight that evidence?"

Core Roles For The First Useful Harness

1

Scout

Collects or extracts data from APIs, local files, search, browser tools, ScrapeCreators, or CAMEL.

InputRaw source: odds page, news article, ranking table, scrape result.
OutputStructured finding with scout name, confidence, and citations.
2

Finding

A compact piece of evidence that the simulation can consume without reading raw web pages.

NowPublic market/stats/odds/news, shared lineup/social, private weather.
LaterBudgets decide who can buy shared or private findings.
3

Predictor

A cheap genome-based participant that interprets findings, forms a probability, listens, and bets or passes.

GenomeWeights, risk appetite, threshold, herd bias, persona.
EvolvesThis is where the genetic algorithm should start.
4

Debater

A predictor selected for this round to publish a compressed public claim.

WhyElite score or wildcard exploration slot.
EffectClaims become a debate finding that all predictors can react to.
Scout -> Finding -> Predictor -> Debater -> Debate Finding -> Bet/Pass

Shared paid access can come later. For now, the useful core is: collect findings, let predictors interpret them, then select a few debaters to make the social finding visible.

Public Debate

Debaters are not separate workers. They are selected predictors for this round.

1. Scouts publish structured findings
2. Predictors make private predictions
3. A few predictors become debaters
4. Debaters publish compressed claims
5. Everyone listens, updates, and bets/pass
All predictors think. Some predictors debate. All predictors decide.

This keeps the debate large enough to matter but small enough to read. It also prevents every predictor from scraping or paying for the same information.

custom API toollow LLM use

Stats Scout

Collects Elo, rankings, recent form, expected goals, or other numeric features. Prefer APIs or local CSVs first.

deterministicCSV/API

News Scout

Uses CAMEL with search/browser tools to find recent news, then returns a compact structured finding.

CAMELcite sources

Lineup Scout

Checks official lineups or injuries. This should be timestamp-gated and source-cited.

browserhigh variance
